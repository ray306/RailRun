## RailRun：更可控的 Agent 流程

大模型 Agent 可以做很多任务，但是在**高确定性场景**下，尤其是**指令长且复杂的任务**里，我们寄希望于让大语言模型“自觉按流程执行”，然而常常遇到以下场景：

- 当前步骤还没做完，Agent 已经开始给结论（*抢跑*）
- 步骤顺序被擅自改写（*流程漂移*）
- 同样输入多次运行，执行路径不一致（*不可复现*）

这里的根本原因是：

*模型越弱，越容易不理解上下文里的流程*;

*模型越聪明，越容易在上下文里自己改写流程*。

如果这些问题会带来明显成本（返工、误判、审计压力、上线风险），你不需要反复调整、测试提示词，然后在另一个模型上重复这些过程，也不需要从 AI 回归传统的程序，而是**执行层面的彻底约束**。

RailRun 就是为了**解决 Agent 在复杂流程中的“脱轨”问题，让执行路径更稳定、可定位、可回溯。**。

---

## 先判断：你现在是否需要 RailRun

更适合立即引入：

- 流程错误代价高（合规、评测、审计、关键交付）
- 需要稳定复跑（同类输入希望路径一致）
- 需要可追溯（出错后必须定位到具体步骤）

可以暂缓：

- 任务以发散探索为主，不追求固定路径
- 错误代价低，复盘价值也低
- 当前瓶颈不是流程失控

---

## RailRun 解决问题的方式

RailRun 的核心思路是：
**把 Agent 当“单步执行器”，而不是“流程自主规划器”。**

> 标准执行循环：
>
> 1. Agent 把流程文件路径交给 RailRun
> 2. RailRun 返回当前唯一可执行步骤
> 3. Agent 执行该步骤
> 4. Agent 回到 RailRun 获取下一步
> 5. 循环直到终态（完成或人工接管）

关键点：

- Agent 看不到完整未来步骤
- 流程推进由外部状态机控制
- 分支基于执行结果推进

RailRun 放弃了“劝模型守规矩”，而在**执行控制面**上减少越权空间。

## 接入成本

你通常只需要：
- 准备一个文件，描述你的流程设计
- （可跳过）参考 `PROTOCOL.md`，用 AI 转译成一个 `arp` 流程描述文件，然后检查
- 在 Agent 中用 `/RailRun <流程文件>` 触发执行

---

## 快速试用

下载并解压：

- [最新安装包（2026.05.17）](https://raw.githubusercontent.com/ray306/RailRun/main/install/railrun2026.05.17.zip)

#### 标准用法：

```
/RailRun {{这里填写流程描述文件的名称（如果不在工作区根目录，请填写完整路径）}}
[如果有补充信息，请写在这里]
```

##### 示例 A：（已安装 Skill）

```text
/RailRun examples\simple_todo_flow.arp
我明天得交论文。
```

##### 示例 B：（未安装 Skill）

```text
请读取 `xxx\railrun\SKILL.md` 并按其中说明执行。
流程文件：examples\simple_todo_flow.arp
补充信息：我明天得交论文。
```

（如果没有被安装为Skill，需要把 `/RailRun` 替换为 `xxx\railrun\SKILL.md`，让 Agent 了解 RailRun 。）

> 试用后你应能直接观察到：
>
> - Agent 严格按照流程走（在其它例子里甚至包括循环和判断）
> - 出错时可定位到具体步骤并回溯重跑

## 进阶：编写流程文件

参考：`PROTOCOL.md` 与 `examples` 目录，自行或借助 AI 把任务流程写成 `arp` 文件。

你也可以用 `md`/`txt` 作为流程输入（`/RailRun {{md/txt 文件路径}}`），RailRun 会先尝试转写再执行。
但关键流程仍建议使用标准 `arp`，以更好地控制流程。

---

## 适用边界与可信度说明

我们承诺的是：在高确定性流程中，RailRun 能提供更强的执行轨道约束。
但 RailRun 当前处于早期阶段，**需要各位用户反馈案例与背书**。

---

## RailRun & Harness

RailRun 可嵌入 Harness 的执行控制层，专门处理“执行确定性”问题。

- Harness：组织并运行 agent 系统
- RailRun：保证系统按轨道运行

如果说 Harness 负责把系统跑起来，那 RailRun 负责让系统始终沿着轨道跑下去。

## **【实验性功能】**

我们提供了实验性方案（需要支持子智能体的环境，如 Codex），通过“流程控制子智能体”向主 Agent 逐步披露步骤。

> 执行流由一个专门的“流程控制子智能体 (Flow Control Subagent)”掌控；
> 主 Agent 处于信息盲区，绝不直接读取任务描述文件；
> 流程由子智能体对主 Agent 进行渐进式披露（每次一步）；
> 主 Agent 每次只执行当前看到的步骤；
> 执行完成后，主 Agent 将执行结果反馈给子智能体，由子智能体智能分析并披露下一步。

Prompt：

```text
请读取 `[EXP] Subagent.md` 的指令，将 {{流程描述文件路径}} 作为参数。
[可选补充信息]
```

目前该方案在某些场景更灵活，但会削弱执行确定性，不建议在高确定性任务中作为默认路径。
