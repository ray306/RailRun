# Agent Rail Protocol

这里描述一种面向 Agent prompt 文件的极简流程协议，用接近 Python 的缩进语法描述：

* 步骤
* 作用域指导
* 条件
* 循环
* 用户提问
* 文件复用
* 函数复用
* 错误中止

文件后缀：

```text
.rail
```

---

# 1. 基本模型

rail 文件由一组语句组成，按顺序执行。

```rail
"""
你是一个研究型 Agent。
回答需要区分事实、判断和不确定性。
"""

理解用户问题
提取用户约束

if user.need_research:
  step:
    搜索资料
    提取关键信息
else:
  直接基于已有上下文回答
```

---

# 2. 缩进

rail 使用缩进表示作用域，类似 Python。

```rail
if user.has_file:
  读取文件
  总结文件
else:
  ask:
    请提供文件
```

规则：

* 同一文件内缩进必须一致。
* 推荐 2 个空格或 4 个空格。
* 控制语句后面的缩进块属于该语句。

---

# 3. Step

## 3.1 隐式 step

和 `step:` 处在同一缩进层级的普通内容（除了函数定义和调用），**每一行都是一个 step**。

```rail
理解用户问题
提取关键约束
生成回答
```

等价于：

```rail
step:
  理解用户问题

step:
  提取关键约束

step:
  生成回答
```

---

## 3.2 显式 `step:`

`step:` 用于把后续缩进块合并成**一个 step**。

```rail
step:
  阅读用户输入
  提取目标、约束和输出格式
  不要立刻回答
```

语义：

* 整个缩进块是一个完整执行步骤。
* 块内多行不会被拆成多个 step。
* 适合表达一个复杂动作。

编译器兼容规则（推荐）：
* 如果 `step:` 块内出现函数调用语句，编译器可自动将该块拆分为“调用前文本 step + 函数调用 + 调用后文本 step”。
* 函数调用应按正常规则展开函数体，不应作为纯文本保留在 step 指令中。
* 建议编译器给出 warning，提示发生了自动拆分。

---

# 4. 作用域整体指导

在任意作用域开头，可以写三引号块：

```rail
"""
你是一个严谨的信息分析助手。
所有回答都要说明不确定性。
"""
```

它表示当前作用域的整体指导。

它也被视为一种特殊 step：**guidance step**。

它和普通 step 的区别是：

* 普通 step：执行一个动作。
* guidance step：更新当前作用域的指导上下文，可以影响后续 step，但 Agent 在这里不执行任何动作。

---

## 4.1 顶层指导

```rail
"""
你是一个研究型 Agent。
不要编造来源。
"""

理解用户问题
生成回答
```

作用于整个文件。

---

## 4.2 函数指导

```rail
def summarize_doc(doc):
  """
  只总结文档内容，不评价立场。
  """

  阅读 {{doc}}
  输出摘要
```

只作用于该函数内部。

---

## 4.3 分支或循环指导

```rail
if user.need_research:
  """
  当前分支必须基于资料回答。
  """

  搜索资料
  综合结论
else:
  """
  当前分支只能基于已有上下文回答。
  """

  直接回答
```

只作用于当前分支。

---

# 5. 注释（仅文档）

所有 `#` 注释都在解析后被忽略，不会被 Agent 看到，也不会进入执行流程。

示例：

```rail
# 这是一条给读者看的说明，不参与执行
理解用户问题
```


---

# 6. 条件控制

```rail
if condition:
  ...
elif other_condition:
  ...
else:
  ...
```

示例：

```rail
if user.has_file:
  读取文件
  总结文件
elif user.has_url:
  打开链接
  提取信息
else:
  ask:
    请提供文件或链接
```

规则：

* 条件语法接近 Python。
* 只执行第一个为真的分支。
* `else` 可选。

---

# 7. 循环

`while`

```rail
while condition:
  ...
```

```rail
break
continue
return
```

语义：

* `break`：退出当前循环。
* `continue`：进入下一轮循环。
* `return`：结束当前函数；如果在顶层使用，则结束整个流程。

示例：

```rail
while True:
  执行下一步
  if 条件满足:
    break
```

建议解释器设置最大循环次数，防止死循环。

`for`

```rail
for item in items:
  ...
```

语义：

* `for` 用于遍历序列。
* 每一轮会暴露“当前元素”和“当前索引”给后续步骤（可用于 `{{item}}` / `{{item_index}}` 模板替换）。
* 迭代是否结束由运行时自动判断，不依赖 Agent 手工回传分支值。

---

# 8. `ask`

`ask` 用于向用户提问，并暂停当前流程，等待用户回答。

```rail
ask:
  你希望最终达成什么目标？
```

语义：

* Agent 向用户提出问题。
* 当前流程暂停。
* 用户回答后，流程从 `ask` 后继续。

---

# 9. 函数

使用 `def` 定义可复用流程块，使用时等同于展开函数体的内容到该位置。但注意：
- {{xxx}}会被替换为参数内容
- 函数签名`def f(a,b):`只是为了方便解析，不会被加入该位置

```rail
def analyze_source(source):
  """
  只分析单个来源，不做综合判断。
  """

  阅读 {{source}}
  提取事实、观点和不确定性
```

调用函数：

```rail
请求用户输入

analyze_source(输入内容)

输出 “分析已完成”
```

多行定义或多行调用（参数跨行）也合法，例如：

```rail
角色扮演(role=孩童🎨, goal='''
  这是一个长文本参数，
  可以跨多行书写。
''')
```

规则：
- 只要函数定义/调用的括号未闭合，后续行都属于同一条定义/调用语句。
- 三引号文本可作为参数值；其中内容按原样传入参数。

结果：

```rail

请求用户输入

"""
只分析单个来源，不做综合判断。
"""

阅读输入内容
提取事实、观点和不确定性

输出 “分析已完成”
```

规则：

* 函数可以有参数。
* 函数体可以包含 step、指导块、条件、循环、ask、函数调用。

---

# 10. 文件复用

`include` 表示原地展开整个文件。

```rail
include "./shared/safety.rail"
include "/prompts/common/research.rail"
```

语义：

> `include path` 等同于把目标文件内容插入当前位置。

示例：

```rail
理解用户问题

include "./shared/safety.rail"

生成最终回答
```

等价于：

```rail
理解用户问题

# ./shared/safety.rail 的内容展开在这里

生成最终回答
```

规则：

* 支持相对路径和绝对路径。
* 相对路径基于当前文件所在目录解析。
* `include` 不创建新作用域。
* 需要检测循环 include。

# 11. 错误模型

rail 默认使用 fail-fast 机制。

> 任意运行错误都会立即中止当前流程，并向用户报告错误位置和原因。

示例错误：

```text
流程执行中止。

错误位置：
  ./main.rail:12

错误原因：
  未定义变量：source
```

如果错误发生在 `include` 文件中：

```text
流程执行中止。

错误位置：
  ./shared/safety.rail:8

引用位置：
  ./main.rail:3

错误原因：
  未定义函数：check_policy
```

常见运行错误包括：

```text
变量未定义
函数未定义
参数数量不匹配
include 文件不存在
include 循环引用
条件表达式无法求值
break / continue 出现在循环外
return 出现在非法位置
ask 恢复失败
```

# 12. 最小语法草案

```text
program     := scope

scope       := (guidance_step | statement)*

guidance_step
            := triple_quote_text

statement   := implicit_step
             | explicit_step
             | ask
             | if
             | while
             | break
             | continue
             | return
             | include
             | def
             | func_call
             | comment

implicit_step
            := text_line

explicit_step
            := "step:" NEWLINE INDENT text_block DEDENT

ask         := "ask:" NEWLINE INDENT text_block DEDENT

if          := "if" expr ":" block
               ("elif" expr ":" block)*
               ("else:" block)?

while       := "while" expr ":" block

def         := "def" name "(" argument_list? ")" ":" block

include     := "include" string_path

func_call   := name "(" argument_list? ")"

argument_list
            := argument ("," argument)*
argument    := value | (name "=" value)

comment     := "#" text_line

block       := NEWLINE INDENT scope DEDENT
```

说明：
- `comment` 会在执行前被忽略。
- `name`、`value`、`expr`、`string_path` 作为词法单元由解释器处理，此处不展开。

容错：
- 条件、参数的写法不做任何限制
- def和func_call里的参数之间同时支持中英文逗号,，
- 各种语法都同时支持中英文冒号:：
- func_call 支持参数跨多行书写（包含三引号长文本）
- 若命名参数值未加引号且包含顶层逗号，编译器可将逗号后的片段并入前一个命名参数值，并给出 warning；建议对含逗号参数值使用引号或三引号，避免歧义
- 若片段无法安全归并，解释器/编译器应直接报错（fail-fast），不应静默改写

# 13. 完整示例

```rail
"""
你是一个研究型 Agent。
所有回答都要区分事实、判断和不确定性。
不要编造来源。
"""

include "./shared/safety.rail"

def clarify_requirement():
  """
  澄清用户需求。
  信息不足时优先提问，不要直接假设。
  """

  if not user.goal:
    ask:
      你希望最终达成什么目标？

  if not user.audience:
    ask:
      这个结果主要给谁看？

  if not user.format:
    ask:
      你希望输出为文档、表格、JSON，还是清单？

def analyze_sources(sources):
  """
  分析多个来源。
  每个来源先独立处理，再统一综合。
  """

  for source in sources:
    # 每个来源单独分析，避免交叉污染
    analyze_source(source)

  synthesize(sources)

clarify_requirement()

if user.sources:
  analyze_sources(user.sources)
else:
  ask:
    请提供需要分析的资料。

生成最终回答
```

# 13. 参数化调用（params）

为了提高 RAIL 流程文件的复用性，RAIL 协议支持在主文件最顶部声明“参数签名”以定义流程的默认参数。

## 13.1 声明默认参数

使用 `params(...)` 在主入口文件的顶部（首个有效行，可有注释和空行在其前）声明默认参数：

```rail
# SKILL.rail
params(no_pause=false, env='dev', max_retries=3)

"""
你是一个研究型 Agent。
"""
理解用户意图
...
```

*   支持的数据类型：布尔值（`true`/`false`）、数字（整数/浮点数）、字符串（单双引号均可）。
*   注意：`params(...)` 是声明在主入口文件的控制标记，在解析时它会被提取并自动移除，不产生任何执行节点。

## 13.2 传参调用

在启动该 RAIL 流程时，可以通过函数调用风格的形式传递参数以覆盖默认值：

```text
# 聊天 Prompt
/RailRun SKILL.rail(no_pause=true)

# 终端 CLI
python next_step.py --procedure "SKILL.rail(no_pause=true, env='prod')"
```

---

# 14. 一句话总结

rail 的核心规则可以压缩成：

> 每行普通内容是一个 step；`step:` 把缩进块合成一个 step；`""" ... """` 是作用域级指导 step；`# ...` 仅供读者阅读且不参与执行；控制流接近 Python；`ask` 暂停等待用户回答；`include` 原地展开文件；`def` 定义可复用流程块；支持 `params(...)` 参数化传参；任意运行错误默认中止并报告位置和原因。

# 16. 从原始流程转写 Rail

> 本节面向执行转写任务的 Agent。当用户提供 `md`/`txt` 文件后，在转写之前必须完整阅读本节。

---

## 16.1 转写的根本原则

**转写不是格式转换，而是意图提炼。**

原始流程（md/txt）是人写给人看的文档，充满了自然语言习惯：段落解释背景、步骤编号表示顺序、列表表示并列、缩进表示从属、粗体强调关键点。Rail 是写给执行引擎看的控制流描述，它的核心是：**谁控制流程推进的时机和条件**。

错误的转写：把原始文档的文字结构"翻译"成 rail 语法结构（一对一映射）。  
正确的转写：理解原始流程的**执行意图**，再用 rail 语法表达最符合该意图的控制流。

---

## 16.2 转写前的阅读与分析（必须先做）

在动手写任何一行 rail 之前，先完整阅读原始文档，回答以下问题：

**① 这个流程的整体目标是什么？**  
用一句话总结。这将成为顶层 `"""..."""` guidance 的核心内容。

**② 流程中哪些地方需要等待用户输入？**  
标记所有"需要用户提供信息才能继续"的位置 → 这些是 `ask:`。

**③ 流程中哪些地方存在"根据条件走不同路径"的分叉？**  
标记所有"如果……则……否则……"的结构 → 这些是 `if/elif/else`。

**④ 哪些步骤需要重复执行？**  
- 明确枚举（对每个文档、对每个来源……）→ `for`  
- 条件反复（直到满足……为止）→ `while`

**⑤ 哪些步骤之间是强顺序依赖（必须串行），哪些只是同一阶段的并列描述？**  
- 强顺序：每步之间有明确的输入输出传递 → 拆分为独立 `step`  
- 并列描述：同一个动作的多个要素 → 合并为一个 `step:` 块

**⑥ 哪些内容是"执行约束/风格要求"而不是"执行步骤"？**  
如"所有输出必须标注来源"、"禁止编造"、"保持严谨风格" → 这些是 `"""..."""` guidance，不是 step。

**⑦ 是否有重复出现的子流程，可以提取为函数？**  
如果同样的逻辑在不同地方出现两次以上 → 考虑 `def`。

---

## 16.3 七步转写流程

按以下顺序操作，不要跳步：

### Step 1：确定顶层 Guidance

把整体目标、角色定位、全局约束写成顶层 `"""..."""` 块。  
**判断标准**：这里的内容是"你是谁、你必须怎么做、整个流程都适用的规则"，而不是"第一步要做什么"。

### Step 2：识别需要用户输入的节点

扫描原始文档，找到所有"在此处需要用户提供信息"的位置，标记为 `ask:` 节点。  
**注意**：`ask:` 意味着流程暂停并等待用户回答。如果是"收集信息然后继续"，才用 `ask:`；如果只是"Agent 自己要思考某个问题"，则是普通 `step:`。

### Step 3：识别条件分支

找到所有分叉路径，用 `if/elif/else` 表达。  
**注意**：原始文档里的"建议"、"可以"、"如果有……"往往对应 `if`；"必须"、"总是"则不是分支，是 step。

### Step 4：识别循环

找到所有重复模式：
- "对每个 X，执行 Y" → `for x in xs:`
- "重复直到条件满足" → `while condition:`
- "反复检查直到用户确认" → `while True:` + 内部 `break`

### Step 5：将剩余步骤归类为 step

剩余的"执行动作"归类：
- 单一、明确的动作 → 隐式 step（单行）
- 多个紧密相关、属于同一逻辑操作的动作 → `step:` 块
- **禁止把不相关的步骤强行塞入一个 `step:` 块**（这会让 Agent 在一步里处理过多内容）

### Step 6：提取可复用函数

如果有两个以上位置出现了相似逻辑，提取为 `def`。  
函数参数用 `{{参数名}}` 标注占位。

### Step 7：自查（见 16.7 节的清单）

---

## 16.4 原始模式 → Rail 构造的识别映射表

| 原始文档中的模式 | 映射到 |
|---|---|
| "目标：…"、"你的角色：…"、"全程必须遵守：…"、"注意：…（全局约束）" | `"""..."""` guidance |
| "请提供…"、"用户需要输入…"、"等待用户选择…" | `ask:` |
| "如果…"、"当…时"、"若…则…" | `if condition:` |
| "否则…"、"如果不满足…" | `else:` / `elif condition:` |
| "对每个…"、"遍历…"、"针对列表中每一项…" | `for item in items:` |
| "重复直到…"、"循环检查…" | `while condition:` |
| "无限循环，直到某条件 break" | `while True:` + `if ...: break` |
| 单个明确的执行动作（一句话） | 隐式 step（单行） |
| 多个紧密相关的子动作（属于同一逻辑阶段） | `step:` 块 |
| 重复出现的子流程 | `def` + 调用 |
| 其它文件包含的共享流程 | `include` |
| "跳过本次"、"进入下一轮" | `continue` |
| "提前退出循环" | `break` |
| "结束函数"、"直接终止整个流程" | `return` |

## 16.5 最常见的转写错误（必读）

### ❌ 错误一：把"说明性文字"当成 step

原始文档：
```text
# 分析阶段
本阶段的目标是对用户提供的数据进行深度分析。
请注意保持客观、不添加主观判断。
```

错误转写：
```rail
# 分析阶段
本阶段的目标是对用户提供的数据进行深度分析。
请注意保持客观、不添加主观判断。
```

正确转写：
```rail
"""
分析阶段：对用户提供的数据进行深度分析。
保持客观，不添加主观判断。
"""
```

> 判断依据：这些文字是"执行约束和背景"，不是"执行动作"。Agent 不需要"执行"它们，只需要在后续执行中遵守它们。用 `"""..."""` guidance 表达。

---

### ❌ 错误二：把列表中的"并列要素"拆成多个独立 step

原始文档：
```text
1. 提取以下字段：
   - 温度
   - 降水
   - 风速
   - 天气码
```

错误转写：
```rail
提取温度
提取降水
提取风速
提取天气码
```

正确转写：
```rail
step:
  提取以下字段：温度、降水、风速、天气码
```

> 判断依据：这四项是同一个"提取动作"的并列内容，不是四个独立动作。拆成四个 step 会让引擎走四轮，而这四项显然是一次操作里完成的。

---

### ❌ 错误三：把"全局约束"分散在每个 step 里重复写

原始文档：
```text
注意：所有输出都要标注来源和日期，禁止编造。

步骤一：搜索资料（注意标注来源）
步骤二：分析数据（注意标注来源）
步骤三：生成报告（注意标注来源）
```

错误转写：
```rail
搜索资料（标注来源和日期，禁止编造）
分析数据（标注来源和日期，禁止编造）
生成报告（标注来源和日期，禁止编造）
```

正确转写：
```rail
"""
所有输出必须标注来源和日期，禁止编造。
"""

搜索资料
分析数据
生成报告
```

> 判断依据：全局约束只需在顶层 guidance 里写一次，它对整个流程中所有后续 step 持续生效。

---

### ❌ 错误四：遗漏条件分支，把分支转成线性 step

原始文档：
```text
如果用户已提供文件，读取文件并总结。
如果用户没有提供文件，向用户索取。
```

错误转写：
```rail
如果用户已提供文件则读取文件并总结
如果用户没有提供文件则向用户索取
```

正确转写：
```rail
if user.has_file:
  读取文件
  总结文件内容
else:
  ask:
    请提供文件
```

> 判断依据：两个"如果"是互斥的执行路径，必须用 `if/else` 表达，否则引擎会把它们都执行一遍。

---

### ❌ 错误五：把"循环"写成重复的 step

原始文档：
```text
对每个来源分别进行分析：
- 来源A：提取关键信息
- 来源B：提取关键信息
- 来源C：提取关键信息
```

错误转写（如果来源是动态的）：
```rail
提取关键信息（来源A）
提取关键信息（来源B）
提取关键信息（来源C）
```

正确转写：
```rail
for source in sources:
  提取 {{source}} 的关键信息
```

> 判断依据：如果来源数量不固定（或是运行时确定的），必须用 `for` 循环，而不是手工展开。手工展开的转写在数量变化时会直接失效。

---

### ❌ 错误六：`ask:` 和普通 step 混淆

原始文档：
```text
需要明确用户的目标。
```

错误转写（如果这实际上需要用户回答）：
```rail
需要明确用户的目标
```

正确转写：
```rail
ask:
  你希望达成什么目标？
```

> 判断依据：区分"Agent 自己思考/判断某件事"（用 `step:`）和"Agent 需要用户主动提供信息才能继续"（用 `ask:`）。若用错，流程要么不会暂停等用户，要么明明 Agent 自己能做却傻等用户。

---

### ❌ 错误七：`if` 条件写成中文描述而非表达式

原始文档：
```text
如果用户没有提供目标，则询问目标。
```

可接受的转写（rail 条件表达式接受自然语言）：
```rail
if not user.goal:
  ask:
    你希望最终达成什么目标？
```

避免的转写（条件过于模糊）：
```rail
if 用户没有提供目标这个信息时应该:
  ask:
    你希望最终达成什么目标？
```

> 判断依据：rail 的 `if` 条件虽然不做严格语法校验，但需要让运行时能够判断真假。条件应当简洁、指向明确的状态（变量名或 Python 风格表达式），不应是长篇叙述。

## 16.6 完整转写示例

**原始 md 文档：**

```text
# 示例任务：写一个今日三项待办

目标：在不调用任何外部工具的前提下，帮助用户产出一个可执行的三项待办清单。

流程：
1. 明确今天最重要目标（1句话）。
2. 拆成三项可执行动作（每项可在30-60分钟内完成）。
3. 为每项动作写开始条件和完成标准。
4. 输出最终清单（编号列表）。

约束：
- 不调用网络、搜索、天气、金融等外部工具。
- 仅基于用户输入与当前对话上下文完成。
- 输出中文。
```

**转写分析过程：**

- "目标"+"约束"段 → 顶层 guidance（全局生效，不是步骤）
- "流程 1-4" → 四个独立且串行的 `step:`（每步的输出是下一步的输入）
- 无条件分支，无循环，无需用户输入（目标假设已知） → 无 `if/while/for/ask`

**转写结果：**

```rail
"""
你是一个待办清单助手。
在不调用任何外部工具的前提下，帮助用户产出一个可执行的今日三项待办清单。
仅基于用户输入与当前对话上下文完成。
输出中文。
"""

step:
  明确今天最重要目标（1句话）

step:
  将该目标拆成三项可执行动作
  确保每项可在30-60分钟内完成

step:
  为每项动作写开始条件和完成标准

step:
  输出最终清单（编号列表）
```

**注意**：约束"不调用外部工具"放在 guidance 里，不在每个 step 里重复写。四个流程步骤之所以都用 `step:` 块而非隐式单行 step，是因为原始文档里每个"步骤"都带有说明性从句（例如"每项可在30-60分钟内完成"），需要保留在指令里。

## 16.7 转写质量自查清单

转写完成后，逐项核对：

**结构层面**
- [ ] 顶层是否有 `"""..."""` guidance 覆盖整体目标和全局约束？
- [ ] 所有"全局约束"是否都在 guidance 里，而不是散布在每个 step 里？
- [ ] 所有"需要用户输入才能继续"的位置是否都用了 `ask:`？
- [ ] 所有互斥的执行路径是否都用了 `if/elif/else` 而非并列 step？
- [ ] 所有重复执行的逻辑是否都用了 `for` 或 `while` 而非手工展开？

**Step 粒度层面**
- [ ] 每个 step 是否只包含"在一次执行中自然完成"的操作？
- [ ] 有没有把同一逻辑动作的并列要素拆成了多个独立 step？
- [ ] 有没有把不相关的步骤强行塞进了一个 `step:` 块？

**条件与循环层面**
- [ ] `if` 条件是否简洁、可判断（不是长篇描述）？
- [ ] `for` 循环的迭代变量命名是否清晰，后续 step 中是否用 `{{变量名}}` 引用了？
- [ ] `while True:` 循环内部是否有明确的 `break` 退出条件？

**整体流程层面**
- [ ] 流程是否有明确的终点（自然结束或显式 `return`）？
- [ ] 原始文档的核心执行意图是否完整保留（没有遗漏关键步骤）？
- [ ] 原始文档里的纯说明性文字是否已转换为 guidance 或注释（`#`），而不是被遗漏或误转为 step？
